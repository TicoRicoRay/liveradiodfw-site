## Root Causes

### 1. Two repos with duplicate site content (PRIMARY)
The site content existed on `gh-pages` branches of BOTH `liveradiodfw-marketing` and `liveradiodfw-site`. This created confusion about:
- Which repo should own the custom domain
- Which repo GitHub Pages should serve from
- Where to push updates (OG Cellars was added to `-site`, other changes went to `-marketing`)

**Why it happened:** The original AI assistant thread (Claude-based) set up the site in the wrong repo (`-marketing`), then content was synced to `-site` as a fix attempt, creating two competing copies.

### 2. GitHub Pages domain lock conflict
GitHub Pages enforces that a custom domain can only be assigned to one repo at a time. The domain was locked to `-site` (which had no CNAME file and wasn't properly configured), while the content being served was in `-marketing`. This created a state where:
- `-marketing` couldn't claim the domain (locked to `-site`)
- `-site` wasn't serving anything useful
- SSL couldn't provision because the domain assignment was in limbo

### 3. Missing GitHub domain verification TXT record
The `_github-pages-challenge-TicoRicoRay` TXT record was not in GoDaddy DNS. Without it, GitHub couldn't verify domain ownership, which blocked domain assignment and SSL provisioning.

### 4. Missing CNAME file on gh-pages branch
The CNAME file (which tells GitHub Pages what domain to serve) was missing from the gh-pages branch at critical moments. It was added to master but not gh-pages, or deleted during troubleshooting attempts.

### 5. Perplexity "Insufficient Credits" bug (EXTERNAL, UNRESOLVED)
The AI assistant thread that was monitoring SSL and building suburb pages hit a false "insufficient credits" error and went completely offline. This meant:
- No SSL monitoring occurred overnight
- No email notification was sent when SSL failed
- No suburb landing pages were built
- Ray had no way to resume that thread for 12+ hours

**Details:**
- Account: rmyers@futurebright.com (Max plan)
- Credits available at time of bug: 40,353 (10,000 Plan + 32,500 Purchased + 39,000 Bonus)
- Ray purchased an additional $150 in credits ($50 × 3) attempting to clear the block
- The purchased credits did not resolve the bug
- The thread remained locked until Perplexity support intervened
- **The $150 in purchased credits should be refunded** (Visa ending 3616)
- As of this writing, the root cause of the credit bug has not been explained by Perplexity support

### 6. Over-reliance on AI assistant for critical monitoring
The overnight SSL monitoring was entirely delegated to the AI thread. When that thread died (due to the credit bug), there was no fallback monitoring, no alerting, and no human check-in scheduled.

## Lessons Learned

1. **One repo, one purpose.** Never have site content in two repos. The canonical site repo is `liveradiodfw-site`. The `-marketing` repo is for automation scripts, email templates, and marketing assets only. Its gh-pages branch should be deleted after this migration is confirmed stable.

2. **GitHub Pages domain setup checklist.** Before going live:
   - [ ] CNAME file exists on the serving branch (not just master)
   - [ ] Custom domain is assigned to exactly one repo in GitHub Pages settings
   - [ ] GitHub domain verification TXT record is in DNS
   - [ ] DNS A/CNAME records point to GitHub's IPs
   - [ ] Wait for SSL cert to actually provision — don't assume it will happen

3. **Don't trust "it'll fix itself" for SSL.** If SSL hasn't provisioned within 30 minutes, something is wrong. Investigate immediately rather than waiting overnight.

4. **Backup monitoring for critical launches.** Don't rely solely on an AI thread for overnight monitoring of a live site. Options:
   - Set up UptimeRobot or similar free monitoring with SMS/email alerts
   - Schedule a manual check-in (alarm on phone) for 1 hour post-launch
   - Have a second team member verify

5. **AI thread continuity is not guaranteed.** Platform bugs, credit issues, or session limits can kill a thread without warning. Critical tasks should not depend on a single AI session surviving overnight.

### Root Causes (Chapter 2)

**1. Silent cert-provisioning stall at GitHub Pages (PRIMARY).**
After the DNS fix on Apr 16 at 5:50 PM, GitHub Pages entered `state: dns_changed` and stayed there for ~14 hours with no internal progress and no error surface. There is no public API to force re-issuance; the only options are the "Retry" button in the web UI (inaccessible on mobile if browser isn't logged in as the repo owner) or a support ticket (multi-day SLA). This is a known failure mode of GitHub Pages that the industry routinely works around by putting a proper edge in front of it.

**2. Original `www` CNAME pointing to apex (CONTRIBUTING).**
The GoDaddy record `www CNAME liveradiodfw.com` is explicitly flagged by GitHub as broken for HTTPS enforcement on a `www` subdomain. GitHub docs require `www CNAME <user>.github.io`. Until this was corrected on Apr 16 at 5:50 PM, cert issuance for www was never going to succeed regardless of anything else.

**3. Misdiagnosis of the `*.github.io` fallback cert as "working as designed" (JARVIS ERROR).**
On Apr 16 afternoon, Jarvis saw `CN=*.github.io` being served on `www.liveradiodfw.com`, knew that GitHub uses this as a fallback for redirector subdomains, and incorrectly concluded that browsers would follow the HTTP 301 transparently. They don't — TLS negotiation happens before HTTP, so every real browser hit the interstitial. The previously-running scheduled monitor was **correct**. Jarvis dismissed it. This extended the outage by the entire Apr 16 afternoon and evening and consumed credits chasing a non-existent problem.

**4. Cardinal rule violations (JARVIS PROCESS ERROR).**
Band-related email lives in Outlook (`info@LiveRadioDFW.com`). Jarvis repeatedly searched the Gmail/gcal connector (which is EOS Worldwide, unrelated to the band) for band email, in violation of a clearly-stated user rule. Additional context was lost and additional credits were wasted as a result.

### Impact (Chapter 2)

- **Site effectively down for www visitors for ~24 hours** (browser cert interstitial blocks the 301 redirect path; users must click through "Advanced → Proceed anyway" to continue — most don't).
- **Apex still worked** — users who typed `liveradiodfw.com` exactly got the site. Anyone from Google search results, old links, or typing `www.` was blocked.
- **Outbound band email continued to work** throughout — MX records stayed pointed at Microsoft 365.
- **Credits consumed on misdirected investigation** — to be added to the refund request.

### Lessons Learned (Chapter 2)

6. **Trust the monitor, not your interpretation of "it's fine."** When a live monitor says a cert is wrong and a browser screenshot confirms it, the monitor is right. Do not theorize your way past evidence.

7. **TLS happens before HTTP.** A 301 redirect cannot rescue a failed TLS handshake. If the cert doesn't match the hostname the client typed, the browser aborts before any HTTP bytes move. This is fundamental and was missed.

8. **GitHub Pages is fine for static content, but its TLS pipeline is a single point of failure with no operator recourse.** For any site that matters, put a proper CDN in front. Cloudflare Free is the obvious choice: it issues certs reliably in 60 seconds, has real status transparency, and gives you the operational knobs GitHub refuses to.

9. **Surgery mindset on DNS changes:** before swapping nameservers, inventory the *full* record set at the current authority (not just web records). Missing an MX or SPF record at nameserver flip = silent email outage. The Cloudflare auto-import covered everything in this case, but verify-then-cut is the rule.

10. **Cardinal user rules are cardinal.** Band/Live Radio DFW work goes to Outlook. EOS is a completely unrelated project that happens to live behind the Gmail connector. Do not cross the streams.
